The Software Factory

Current-state architecture note for the local Software Factory workflow engine: a SwiftUI macOS client driving a Bun/TypeScript daemon over WebSockets, with shared Zod schemas acting as the protocol boundary and append-only session logs as the durable substrate.

Client

SwiftUI desktop app in apps/mac. It renders sessions, graphs, transcripts, role/workflow editors, OpenAI auth state, and workspace launch actions.

Daemon

Bun-first TypeScript service in apps/daemon, with a Node fallback. It owns sessions, workflow execution, auth, event storage, workspace coordination, and OpenAI agent runtime calls.

Protocol

packages/shared defines the Zod contract for requests, responses, notifications, events, roles, workflows, snapshots, and graph state.

What changed since the first pass

Workflow/role surface became first-class

The client now has menu entries for Roles and Workflows, can edit role definitions, and can instantiate workflow templates into an active session graph. The daemon backs this with listRoles, upsertRole, listWorkflows, and instantiateWorkflow.

Auth moved from placeholder to operational seam

The server exposes OpenAI OAuth lifecycle endpoints through WebSocket methods plus an HTTP callback route. Non-debug live agent runs use a connected Codex OAuth token against WHAM, or an OpenAI API key from Settings/OPENAI_API_KEY.

Runtime now has a real OpenAI Agents path

OpenAIAgentRuntime builds role-specific agents, injects workflow tools, runs with a bounded turn count, and emits structured session events. Debug mode still routes to a deterministic runtime.

Workspace/event model got heavier

The daemon initializes per-session workspaces, tracks file claims/touches/conflicts, stores append-only JSONL event streams, rebuilds snapshots, and tracks active tool calls from tool_call/tool_result event pairs.

Client architecture

The Swift app is a shell around SessionStore. Views are fairly thin: they select/edit state, then call store actions which emit WebSocket requests.

  • Navigation: ContentView switches between sessions, roles, and workflows.
  • Connection: DaemonClient manages URLSessionWebSocketTask.
  • State: SessionStore tracks sessions, selected agent, graph, transcript, roles, workflows, auth, debug mode, and workspace root.
  • UI affordances: open workspace in VS Code/Finder/iTerm; pause/resume/cancel selected agent; instantiate workflows; begin/disconnect OAuth.

Server architecture

The daemon is a local control plane. It validates requests through shared schemas, mutates append-only event logs, rebuilds snapshots, and fans events back to subscribers.

  • Transport: Bun Bun.serve on port 3767, with Node fallback using ws.
  • Core: SessionManager owns request handling and workflow activation.
  • Runtime: OpenAI Agents SDK for live mode; deterministic runtime for debug mode.
  • Persistence: per-session events.jsonl, per-agent transcript.jsonl, and snapshot.json.

Workflow model

Workflows are declarative graph specs. Each spec names roles, nodes, edges, concurrency, lifecycle, and stop criteria. Built-ins currently include:

  • planner-orchestrator: orchestrator asks planner for workflow details, planner replies.
  • implementor-reviewer: orchestrator hands to implementor; implementor and reviewer exchange messages.
  • implementor-qa-loop: implementor hands to QA; QA sends issues/acceptance back.

Current constraints / sharp edges

  • The repo still presents itself as a local alpha; no release artifacts.
  • Auth refresh appears deliberately conservative: if OAuth needs refresh, live runs ask the user to reconnect.
  • The WebSocket protocol is simple JSON envelopes, not a formal JSON-RPC implementation.
  • Workflow execution is bounded and deterministic-ish at the graph-scheduler level, but live agent output is obviously non-deterministic.
  • The Swift model surface appears to be evolving; role editor references canCreatePlans, while the shared schema shown here defines read/write/run-command policy.

Operational mental model

Client intent
  → WebSocket JSON request
    → Zod validation
      → SessionManager command
        → EventStore append
          → snapshot rebuild
            → WebSocket event notification
              → SessionStore applies event
                → SwiftUI graph/transcript rerender

The architecture is basically Event Sourcing: Bedroom Edition. The client is not the authority; it’s a fast, optimistic retina over the daemon’s log-shaped soul.
